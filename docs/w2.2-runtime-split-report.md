## Path Decision: Path A (Pre-Built Scene Helper)

The task brief offered two paths for the GLTFExporter retirement:

- **Path A**: Extract `renderSceneToGLB(scene, clips?)` helper in
  `core/kiln/render.ts`, callable from both headless `renderGLB(code)` and
  the editor's live runtime.
- **Path B**: Pass the user's code string from editor back through
  `core.kiln.renderGLB(code)`, re-executing headlessly. Simpler but slower
  (re-compiles + re-renders).

**Chose Path A.** It's the right factoring (one bridge, two callers) and
the editor avoids re-execution overhead. The implementation also enabled
swapping `NodeIO` for `WebIO` across both paths - WebIO works in Node too
because `writeBinary()` is pure bytes-out, never touching the abstract
`readURI`/`resolve`/`dirname` methods that differentiate the IO classes.
This means there is now exactly one IO class in the GLB pipeline.

### Browser-safety of the core import

The new editor `export.ts` imports from `@pixel-forge/core/kiln/render`,
not from `@pixel-forge/core/kiln` (the index barrel). The barrel pulls
`generate.ts`, which imports `@anthropic-ai/claude-agent-sdk`. By
importing the dedicated subpath, the browser bundle stays clean - it pulls
only `render.ts -> primitives.ts + @gltf-transform/core` (already a
client-safe dep).

The `WebIO` source module re-exports `NodeIO` alongside it from
`@gltf-transform/core/dist/io/index.js`, but Vite's tree-shaker drops
NodeIO since it isn't referenced. NodeIO would only pull `node:fs`/`path`
on `.init()`, which is never called.

The `./kiln/render` subpath was added to `packages/core/package.json`
exports.

## Surprises in the Split

1. **`new Blob([Uint8Array])` typing on Web TS lib.** The browser's `Blob`
   constructor types `BlobPart` as `ArrayBuffer | ArrayBufferView<ArrayBuffer>
   | string`, but `gltf-transform`'s `WebIO.writeBinary()` returns
   `Uint8Array<ArrayBufferLike>`, where `ArrayBufferLike` could in
   principle be `SharedArrayBuffer`. TypeScript rejects the assignment.
   Fix: slice into a fresh `ArrayBuffer` before blob construction. This is
   a pure type fix - the runtime bytes are identical.

2. **`renderSceneToGLB` works in Node too.** `WebIO.writeBinary()` only
   serializes; it never calls the abstract `readURI`/`resolve`/`dirname`
   methods that differentiate the IO classes. So `WebIO` in Node is fine
   for the write side, and we get one bridge for both contexts. Replacing
   `NodeIO` with `WebIO` in the headless `renderGLB` path also worked with
   no test regressions.

3. **State sharing between sub-modules.** The 8-way split called for
   per-concern files, but the runtime fields (renderer, scene, camera,
   controls, mixer, etc.) cross every concern. Solution: a `state.ts`
   module exporting a `RuntimeState` interface, and each sub-module
   exports functions that take `state: RuntimeState` as the first
   parameter. The class in `runtime.ts` stays a thin orchestrator that
   instantiates the state and forwards calls. This pattern keeps each
   sub-module testable in isolation while preserving the original
   `KilnRuntime` class API.

4. **`templates.ts` deserved its own file.** The original file mixed the
   `KilnRuntime` class with the `generateStarterCode()` template-string
   builder for new asset starter code. They're orthogonal concerns - the
   templates are pure string functions with zero runtime dependencies.
   Pulled them into `templates.ts` (167 LoC of which is the four template
   string literals).

5. **`animate()` was a class field arrow function.** The original used
   `private animate = (): void => {...}` so it could be passed directly
   to `requestAnimationFrame`. After extraction, the equivalent pattern
   is a local `step` function inside the exported `animate(state)` -
   functionally identical, idiomatic without the class context.

## Path NOT Taken

- **Path B** (pass code string back to core, re-execute headlessly)
  rejected because (a) Path A is the cleaner factoring, (b) the editor
  already has the executed scene in memory; re-executing wastes 50-150ms
  per export, and (c) keeping the editor's path unique to the live scene
  preserves the option for the editor to export user-edited scene state
  (e.g., picked color, repositioned parts) without round-tripping through
  code regeneration.

- **Coverage tests for each sub-module.** Out of scope for W2.2 (split
  is a refactor, behavior preserved); but the new sub-module shape
  (`fn(state, ...args)` instead of `this.fn(...)`) makes future per-
  module tests trivial. Logged as W7.3 follow-up territory.
