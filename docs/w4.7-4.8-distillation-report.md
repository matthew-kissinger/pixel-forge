### Recipe shape

Every recipe follows the same 5-line core:

```ts
import { image, providers } from '@pixel-forge/core';

const pipeline = image.pipelines.createSpritePipeline({
  imageProvider: providers.createGeminiProvider(),
  bgRemovalProvider: providers.createFalBgRemovalProvider(),
});

const result = await pipeline.run({ prompt, refs, ... });
writeFileSync(outPath, result.image);
```

Anything past that is just per-recipe data (prompts, asset names,
output paths). All chroma cleanup, BiRefNet, retry-on-rate-limit,
timeouts, error translation, and resume-on-disk-skip live in core.

## Decisions

### Python vs TS for GLB scripts

The audit calls out Python as "the right choice for GLB/Kiln batches
that POST to localhost:3000." The new approach lets us call core
directly from TS via `image.pipelines.createGlbPipeline()`, which
dissolves the urllib + subprocess + JSON dance entirely. So:

- **`gen-weapons.ts` (new)** — converted from Python to demonstrate the
  pattern. 167 lines → 134 lines, no running server required, batch
  wrapper handles resume.
- **5 Python GLB scripts kept as-is** — `gen-buildings.py`,
  `gen-ground-vehicles.py`, `gen-aircraft-watercraft.py`,
  `gen-firebase.py`, `gen-animals-v2.py`, `gen-remaining.py`. They
  work today, and converting them to TS is mechanical busywork that
  can happen incrementally as each batch is next-touched.

**Principle for future work:** new GLB recipe scripts should start as
TS using the glb pipeline; existing Python scripts stay until they
need a substantive change.

### `gen-ui-icons.ts` kept its CLI shape

This is the most user-facing recipe (sheet/seed/batch/run/redo/list
subcommands appear in CLAUDE.md and `docs/`). Rewriting it to drop
the CLI would have broken muscle memory for no value. Internals were
collapsed (chroma + variant + retry → icon pipeline) but the public
surface is identical: `bun scripts/gen-ui-icons.ts seed weapons` still
works exactly as before.

The style sheet generation step bypasses the icon pipeline directly
(`imageProvider.generate(...)` rather than the full pipeline) so the
magenta background is preserved — the sheet is a downstream reference
image, and stripping its bg would weaken the visual signal Gemini reads
from later "match this style" prompts.

### Type imports through `image.pipelines.<TypeName>`

The recipe pattern reaches into `image.pipelines.SpriteOutput`,
`TextureOutput`, `GlbOutput` for batch wrapper generics. These work
fine because the namespace re-export in `packages/core/src/image/index.ts`
includes type re-exports. No changes to core needed.

## Pipeline gaps discovered

None blocking. Two minor observations during the rewrite:

1. **No way to opt out of style suffix entirely** in the icon pipeline.
   `styleOverride: ''` works (the pipeline does `${prompt}, ${style}`,
   so an empty style yields a trailing `, ` which Gemini ignores), but
   a `styleOverride: null` or boolean opt-out would be cleaner. Not
   urgent — only affects the style-sheet generation, which I worked
   around by going through the provider directly.

2. **Batch wrapper requires the inner pipeline to take a single typed
   input.** Recipes that adapt a local `Asset` shape to the pipeline's
   `SpriteInput` build a tiny anonymous `Pipeline<Asset, SpriteOutput>`
   wrapper. This works but is a 5-line ritual repeated in every recipe.
   A `createBatchPipeline({ pipeline, mapInput, getOutputPath })`
   convenience could remove the wrapper. Defer until we have 5+ recipes
   doing this — currently we have 4 (vegetation-redo, batch1-v2,
   batch2-redo, textures-v3, weapons). Worth proposing in W6 if the
   pattern persists.

3. **`createSoldierSetPipeline` always regenerates the T-pose.** The
   `gen-arvn-remaining.ts` recipe had to bypass it and call the sprite
   pipeline directly to reuse a pre-generated T-pose from disk. A
   `tPose: Buffer | { prompt, refs? }` discriminated input would let
   resumed soldier sets skip the most expensive call. Not urgent, but
   would simplify partial-regen flows.

## Validation

I did **not** run a recipe end-to-end against live APIs to verify
visual output. Reasoning per the task instructions:

- Skipping live runs avoids API spend on a refactor that doesn't change
  any prompt or pipeline logic — the recipes call into core paths
  already covered by 85+ pipeline unit tests.
- typecheck + lint + 1931 client / 114 server / 157 core tests passing
  is sufficient signal that no import is broken and no behavior shifted.
- The recipe shapes are direct ports of pre-existing scripts that have
  produced production assets in `war-assets/`. The only behavioral
  change is that retry / chroma / bg-removal now go through core's
  hardened path rather than ad-hoc per-script implementations — and
  the core path is what 85 pipeline tests exercise.

If a regression surfaces during the next live regen run, restore from
git history (`refactor/scripts-distill` commit before the recipe
commit) and report the diff.
