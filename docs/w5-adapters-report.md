# W5 — CLI + MCP Adapters Report

**Status:** Complete
**Date:** 2026-04-21
**Scope:** W5.1 (`packages/cli`) + W5.2 (`packages/mcp`)

## Summary

Two thin adapters over `@pixel-forge/core` so agents can drive the same
substrate as the editor. Both adapters: zero core changes, parallel
shape, identical error semantics.

---

## W5.1 — CLI (citty)

Package: `packages/cli` · binary: `pixelforge`

### Commands implemented

```
pixelforge gen sprite       --prompt --bg --refs --out [--no-birefnet] [--preserve-flash]
pixelforge gen icon         --prompt --variant --refs --out
pixelforge gen texture      --description --size --out
pixelforge gen glb          --prompt --category --style --out [--save-code] [--no-animation]
pixelforge gen soldier-set  --faction --tpose-prompt --poses-file --out-dir --bg [--style-refs]
pixelforge inspect glb      <file>
pixelforge providers list
pixelforge providers pick   --kind --refs --transparency [--prefer-cheap] [--prefer-provider]
pixelforge kiln list-primitives [--category]
pixelforge kiln validate    <file>
pixelforge kiln inspect     <file>
pixelforge kiln refactor    --code --instruction --out
```

Every command accepts `--json`. Errors print `code` + `fixHint` to stderr,
exit non-zero. Help is auto-generated by citty.

### Notable design choices

- **Provider loader** (`src/providers.ts`) is a single helper called by
  every command. Builds providers from env vars; missing keys surface as
  `ProviderAuthFailed` with the env-var name in the message. Image
  routing prefers OpenAI when `OPENAI_API_KEY` is present (better
  multi-ref fidelity), else Gemini.
- **Routing helper** (`src/routing.ts`) mirrors core's `pickProviderFor()`.
  Necessary because core's `index.ts` only re-exports `capabilities()`
  on the public namespace — `pickProviderFor` lives in the same module
  but isn't surfaced. Reaching through the source path is blocked by
  core's restrictive `exports` map. The helper walks `capabilities()`
  using the same rules. **Followup**: surface `pickProviderFor` from
  `@pixel-forge/core`'s public index in a future core change so the
  duplication can drop.
- **Provider type imports**: types come from the `providers` namespace
  (e.g. `providers.ImageProvider`), not as bare top-level types. Trips
  every adapter consumer; worth adding a type re-export to core.

## W5.2 — MCP server (stdio)

Package: `packages/mcp` · binary: `pixelforge-mcp`

### Tools registered

| Tool | Returns |
|------|---------|
| `pixelforge_gen_sprite` | `{ path, sizeBytes, meta }` (or base64 with `inline:true`) |
| `pixelforge_gen_icon` | same |
| `pixelforge_gen_texture` | same |
| `pixelforge_gen_glb` | same + `code` |
| `pixelforge_gen_soldier_set` | `{ tposePath, posePaths[], meta }` (always disk) |
| `pixelforge_kiln_inspect` | `InspectResult` (tris, bounds, named parts, animations) |
| `pixelforge_kiln_validate` | `ValidationResult` (issues + warnings) |
| `pixelforge_kiln_refactor` | `{ code, usage }` |
| `pixelforge_kiln_list_primitives` | `{ count, primitives[] }` (optional category filter) |
| `pixelforge_providers_capabilities` | `{ providers[] }` (full matrix) |

Server boots via `StdioServerTransport`, registers tools, prints one
startup line to stderr. Install with:

```bash
claude mcp add pixelforge --stdio bun packages/mcp/src/index.ts
```

### Output strategy for binary payloads

**Default: write to a tmp path, return `{ path, sizeBytes, meta }`.** Agents
read the file with their own filesystem tools. This keeps every MCP
message under a few KB even when the asset is megabytes.

**Opt-in inline base64**: pass `inline: true` (or `outPath: "..."` for an
explicit destination) on any of the 4 binary-producing tools. `outPath`
respects the user's directory choice; tmp path uses
`os.tmpdir()/pixelforge-<rand>/<uuid>.<ext>`.

`pixelforge_gen_soldier_set` always writes to disk because it produces
N+1 files; the structured response lists every path.

### Error handling

Caught `PixelForgeError` instances surface as
`{ ok: false, code, message, fixHint, retryable }` in `structuredContent`
with `isError: true`. Non-Pixel-Forge errors map to a generic
`UNKNOWN_ERROR` envelope. Agents read `fixHint` and can act on
`retryable` directly.

## Surprises & lessons (citty + MCP SDK in 2026)

1. **`@modelcontextprotocol/sdk` v1.29 is still the production line on
   npm.** v2.0.0-alpha.2 split into `@modelcontextprotocol/{server,client,node}`
   exists but is alpha. Stuck with v1 imports
   (`@modelcontextprotocol/sdk/server/mcp.js`,
   `@modelcontextprotocol/sdk/server/stdio.js`,
   `@modelcontextprotocol/sdk/inMemory.js`).
2. **citty 0.2.2 lazy subcommands** can be either a function returning a
   command or the command itself. Tests need to handle both shapes
   (`typeof sub === 'function' ? sub() : sub`) — no helper for it.
3. **`exports` map in core blocks deep imports** even within the
   workspace. Could not import `pickProviderFor` from
   `@pixel-forge/core/src/capabilities.ts`. Mitigated by reproducing the
   logic in `cli/src/routing.ts`. Cleaner long-term: add to core's
   public index.
4. **Bun auto-loads `.env.local`**, so neither adapter needs a dotenv
   shim. Worth documenting; saved one round-trip per command.
5. **MCP `registerTool` zod schema goes inside `inputSchema:`**, not as
   a positional second arg. Matches `registerPrompt`/`registerResource`
   shape.

## Deferred

- **`pickProviderFor` re-export from core** — cross-cutting; out of W5
  scope, opens a small core change for W6 or a follow-up cycle.
- **Live integration tests for `gen_*`** — every adapter test today is
  pure (no network). A `CLI_LIVE=1` / `MCP_LIVE=1` gated suite that
  generates one sprite and inspects the output bytes would cover the
  end-to-end path without burning the budget.
- **`bun link` automation** — manual step today; could be wired into a
  root `bun run install:cli` if it gets annoying.
- **Shape parity check between CLI flags and MCP tool inputs** — both
  hand-authored from the same core schemas; a generative test could
  catch drift.
